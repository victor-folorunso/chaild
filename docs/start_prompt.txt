SQL PATCH RULE (non-negotiable):
  NEVER modify an existing SQL patch file.
  If a bug requires a schema change, CREATE A NEW PATCH with the next
  available number (e.g. if 009 exists, create 010_fix_description.sql).
  The user runs patches incrementally in Supabase SQL Editor. Editing an
  already-run patch would have no effect and break the audit trail.

DELETE ACCOUNT SAFETY RULE:
  deleteAccount() calls the existing rpc('delete_user') which runs with
  SECURITY DEFINER and deletes the auth.users row. Supabase cascades this
  to all tables with user_id foreign keys. This is safe for the Chaild
  platform — it removes the user's data cleanly.
  The concern is partner-side: if a partner calls ChaildAuth.initialize()
  and exposes account_screen.dart, end users of THAT partner app can delete
  their Chaild account (which is their account across ALL partner apps).
  This is intentional product behaviour — it is a shared account SDK.
  Do NOT add extra guards or confirmation flows beyond what already exists
  in account_screen.dart unless explicitly instructed.

==============================
HOW TO WORK: THE EDIT LOOP
==============================

This is the core working pattern. Follow it for every single task.

BEFORE starting a task:
  git -C "C:\Users\victo\Documents\SandBox\chaild" diff --stat HEAD
  -> Must be empty. If not, finish or revert the previous task first.

TO READ an existing file you have not seen this session:
  Read it once. Remember it. Never read it again in the same session.
  Use git diff to see changes to it after that first read.

TO MODIFY an existing file:
  Use edit_block with a surgical old_string/new_string. Never rewrite the
  whole file. Include only the minimum surrounding lines needed to make the
  old_string unique. 3 lines of context is usually enough.

TO CREATE a new file:
  Write it in chunks if it is over 30 lines. Append subsequent chunks.
  Do not read it back after writing unless the write tool reports an error.

AFTER completing a task (one logical unit of work):
  1. Run: git -C "C:\Users\victo\Documents\SandBox\chaild" diff --stat HEAD
     Verify: every file that should have changed is listed.
     Verify: no unexpected files appear.
     Verify: line count deltas look plausible (a new 80-line service file
     should show +80, not +5 — +5 means a partial write happened).

  2. If anything looks wrong, run:
     git -C "C:\Users\victo\Documents\SandBox\chaild" diff HEAD -- path/to/file
     Read only that file's delta to diagnose the problem. Fix it.

  3. When the diff stat is clean and correct, commit:
     git -C "C:\Users\victo\Documents\SandBox\chaild" add -A
     git -C "C:\Users\victo\Documents\SandBox\chaild" commit -m "Fix: brief description"

     Use "Fix: ..." as the commit message prefix for bug fix commits.
     Use section number prefixes for roadmap work, e.g.:
       "Section 2.3: add 005_rpc_delete_user.sql patch"

  4. Move to the next task. The commit is the checkpoint. If this session
     ends before the next commit, the next session will find it via
     git diff --stat HEAD and know exactly what to finish.

NEVER commit a partial task. A commit means the task is complete and the
code compiles (or the SQL/TS file is syntactically valid).
