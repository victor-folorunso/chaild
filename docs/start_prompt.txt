STEP 0: ORIENT YOURSELF FIRST
==============================

Before reading any source file, run these two git commands. They cost almost
no tokens and tell you everything about the current state of the repo.

  git -C "C:\Users\victo\Documents\SandBox\chaild" log --oneline -10

This shows the last 10 commits. Each commit message names the section that
was completed. The most recent commit tells you exactly where to resume.

  git -C "C:\Users\victo\Documents\SandBox\chaild" diff --stat HEAD

This shows any work-in-progress that was not committed yet (partial writes,
incomplete tasks). If this is empty, everything is clean and committed.

DECISION TREE:

A) No commits exist yet (fresh repo):
   -> Read all docs (see DOCS TO READ below), then read existing source files
      listed under WHAT ALREADY EXISTS, then begin Section 2.

B) Commits exist, diff --stat HEAD is empty (clean between sessions):
   -> Read only the commit log to find the last completed section.
   -> Run: git diff HEAD~1 --stat
      This shows what the last session changed. Read only those specific files
      using: git diff HEAD~1 -- path/to/file
      This gives you the delta only, not the whole file. Use it to reconstruct
      context for the next task without reading entire files.
   -> Resume at the next incomplete section.

C) Commits exist, diff --stat HEAD is NOT empty (session ended mid-task):
   -> Run: git diff HEAD -- path/to/each/modified/file
      Read each delta to understand what was partially done.
   -> Either complete the partial work or revert it cleanly with:
      git checkout HEAD -- path/to/file
   -> Then resume from that task.

NEVER read a whole source file if you can reconstruct its relevant context
from git diff output. A diff of 20 lines beats reading 300 lines every time.

==============================
HOW TO WORK: THE EDIT LOOP
==============================

This is the core working pattern. Follow it for every single task.

BEFORE starting a task:
  git -C "C:\Users\victo\Documents\SandBox\chaild" diff --stat HEAD
  -> Must be empty. If not, finish or revert the previous task first.

TO READ an existing file you have not seen this session:
  Read it once. Remember it. Never read it again in the same session.
  Use git diff to see changes to it after that first read.

TO MODIFY an existing file:
  Use edit_block with a surgical old_string/new_string. Never rewrite the
  whole file. Include only the minimum surrounding lines needed to make the
  old_string unique. 3 lines of context is usually enough.

TO CREATE a new file:
  Write it in chunks if it is over 30 lines. Append subsequent chunks.
  Do not read it back after writing unless the write tool reports an error.

AFTER completing a task (one logical unit of work):
  1. Run: git -C "C:\Users\victo\Documents\SandBox\chaild" diff --stat HEAD
     Verify: every file that should have changed is listed.
     Verify: no unexpected files appear.
     Verify: line count deltas look plausible (a new 80-line service file
     should show +80, not +5 — +5 means a partial write happened).

  2. If anything looks wrong, run:
     git -C "C:\Users\victo\Documents\SandBox\chaild" diff HEAD -- path/to/file
     Read only that file's delta to diagnose the problem. Fix it.

  3. When the diff stat is clean and correct, commit:
     git -C "C:\Users\victo\Documents\SandBox\chaild" add -A
     git -C "C:\Users\victo\Documents\SandBox\chaild" commit -m "Section X.Y: brief description"

     Use the section number from ROADMAP.md as the commit message prefix.
     Example: "Section 2.3: add 005_rpc_delete_user.sql patch"
     Example: "Section 5.1: scaffold chaild_storage package"

  4. Move to the next task. The commit is the checkpoint. If this session
     ends before the next commit, the next session will find it via
     git diff --stat HEAD and know exactly what to finish.

NEVER commit a partial task. A commit means the task is complete and the
code compiles (or the SQL/TS file is syntactically valid).
